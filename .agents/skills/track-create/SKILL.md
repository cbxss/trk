# Track-Create Skill

Generate a domain-specific tracking skill for this project. Works like `autoresearch-create` but for generic work tracking.

## Setup Questions

Ask (or infer from context):

1. **Domain** — What type of work? Examples:
   - `security-research` — vulnerability hunting, exploit development
   - `data-migration` — schema changes, backfills, rollbacks
   - `debugging` — root cause analysis, reproduction attempts
   - `refactoring` — code cleanup, performance optimization
   - `feature-dev` — product features, incremental rollout
   
2. **Target** — What's being tracked?
   - For security-research: `doordash`, `uber-bug-bounty`
   - For data-migration: `production_users_db`, `billing_schema`
   - For debugging: `heap-corruption-bug`, `race-condition-login`

3. **Terminology** — Domain-specific names:
   - **Work item** (default: "work-item", examples: hypothesis, migration, bug-theory, task)
   - **Confirmation** (default: "confirmation", examples: vulnerability, applied-migration, root-cause, completed-feature)
   - **Attempt** (default: "attempt", examples: payload, validation-run, reproduction-step, test)

4. **Status Progression** — How confirmations evolve:
   - security-research: `confirmed → confirmed_needs_exfil → confirmed_needs_poc → submitted → bounty_paid`
   - data-migration: `planned → testing → validated → applied → rolled_back`
   - debugging: `investigating → reproduced → fixed → verified`

5. **Files in Scope** — What can the agent modify?
   - List key files/directories relevant to the domain
   
6. **Off Limits** — What must NOT be touched?
   - Usually: `node_modules/`, `.git/`, production config, etc.

## Generate Skill File

Write `.agents/skills/<domain>/SKILL.md`:

```markdown
---
name: <domain>
description: <Domain-specific tracking for target>. Generated by track-create.
---

# <Domain> Tracking

## Target
<target-name>

## Vocabulary
- **Work Item** = <work-item-term> (<what it represents>)
- **Confirmation** = <confirmation-term> (<what it represents>)
- **Attempt** = <attempt-term> (<what it represents>)

## Status Progression
<status progression for confirmations>

## Commands (with domain meaning)

### Core Workflow
- `track init <target>` — Initialize tracking for this target
- `track show` — View current state (always start sessions with this)
- `track new --desc "..." [--priority N] [--next "..."]` — Create <work-item-term>
- `track try <ID> "<attempt>" --result "..."` — Record <attempt-term>
- `track close <ID> --conclusion "..."` — Close <work-item-term> (didn't work out)
- `track confirm <ID> --conclusion "..." [--title "..."] [--severity <level>]` — Create <confirmation-term>

### Search & Update
- `track grep "<pattern>"` — Search all content (check before trying anything new)
- `track update <ID> --next "..." [--priority N]` — Update work item
- `track reopen <ID>` — Reopen closed work item

### Queue Pattern (enumerate before drilling)
- `track qadd --desc "..." [--queue <name>]` — Add item to work queue
- `track qbulk --queue <name>` — Bulk-add from stdin
- `track qdone <QID> [--hyp <ID>]` — Mark queue item done
- `track qshow [--queue <name>]` — View queues

### Notes & State
- `track note "observation"` — Record timestamped observation
- `track last "action"` — Update last action taken
- `track block "reason"` — Mark as blocked
- `track unblock` — Clear block

## Domain-Specific Patterns

### <Pattern 1>
<Describe common workflow for this domain>

Example:
\`\`\`bash
track new --desc "..."
track try <ID> "..."
track confirm <ID> --title "..."
\`\`\`

### <Pattern 2>
<Another common pattern>

## Files in Scope
<List of files/directories the agent may modify>

## Off Limits
<What must NOT be touched>

## Rules
1. Always run `track show` at session start
2. Use `track grep` before trying anything (avoid duplicate work)
3. Record every <attempt-term> with `track try`
4. Use queues to enumerate all items before drilling into any one
5. Close dead ends immediately with `track close`
6. Confirm successes with `track confirm`
7. Update `last_action` after significant steps

## What's Been Tried
<Update this section as work progresses — key wins, dead ends, patterns>

(Agent updates this section periodically)
```

## Initialize and Start

After generating the skill:

1. Commit the skill file
2. `track init <target>`
3. `track show` — verify
4. Start working with domain awareness

The agent now has access to this project-local skill and will use domain terminology automatically.

## Example Sessions

### Security Research
```bash
User: "start tracking this security research"

Agent (asks):
> Domain? security-research
> Target? doordash
> Work item term? hypothesis
> Confirmation term? vulnerability
> Status progression? confirmed → confirmed_needs_exfil → submitted → bounty_paid

Agent (generates .agents/skills/security-research/SKILL.md):
> Created security research tracking skill
> Initialized tracking for doordash

Agent: "Let's start. What should I investigate first?"
User: "check postMessage handlers"
Agent: track new --desc "XSS via postMessage" --priority 1
       track qadd --queue listeners --desc "chunk_abc.js L123 navigate handler"
       ...
```

### Data Migration
```bash
User: "track this database migration"

Agent (asks):
> Domain? data-migration
> Target? production_users_db
> Work item term? migration
> Confirmation term? applied-migration
> Status progression? planned → testing → validated → applied

Agent (generates .agents/skills/data-migration/SKILL.md):
> Created data migration tracking skill
> Initialized tracking for production_users_db

Agent: "What migrations do we need?"
User: "add user_preferences table"
Agent: track new --desc "Add user_preferences table"
       track try M1 "CREATE TABLE..." --result "foreign key constraint error"
       ...
```

## Notes

- The generated skill is **project-local** (`.agents/skills/<domain>/`)
- Pi loads project-local skills automatically when in that directory
- The skill teaches the agent domain-specific vocabulary and workflows
- Each project can have its own customized domain skill
- The `track` CLI remains 100% generic and domain-agnostic

# Track-Create Skill

Generate a domain-specific tracking skill for this project.

## Setup Questions

Ask (or infer from context):

1. **Domain** — What type of work?
   - `security-research`, `data-migration`, `debugging`, `refactoring`, `feature-dev`
   
2. **Target** — What's being tracked?
   - For security: `doordash`, `uber-bounty`
   - For migration: `production_users_db`, `billing_schema`
   - For debugging: `heap-corruption-bug`, `race-condition`

3. **Work item term** — What do you call tasks?
   - Examples: hypothesis, migration, bug-theory, task, feature

4. **Confirmation term** — What's a successful completion?
   - Examples: vulnerability, applied-migration, root-cause, deployed-feature

5. **Attempt term** — What's each try called?
   - Examples: payload, validation-run, reproduction-step, test

6. **Status progression** — How do confirmations evolve?
   - security: `confirmed → needs_exfil → needs_poc → submitted → bounty_paid`
   - migration: `planned → testing → validated → applied → rolled_back`
   - debugging: `investigating → reproduced → fixed → verified`

7. **Files in scope** — What can be modified?

8. **Off limits** — What must NOT be touched?

## Generate Project-Local Skill

Create `.agents/skills/<domain>/SKILL.md`:

```markdown
---
name: <domain>
description: <Domain> tracking for <target>. Generated by trk-create.
---

# <Domain> Tracking: <target>

## Vocabulary

- **Work Item** = <work-item-term> (<brief description>)
- **Confirmation** = <confirmation-term> (<brief description>)
- **Attempt** = <attempt-term> (<brief description>)

## Status Progression

<confirmation status flow>

## Track Commands (with domain meaning)

### Session Start
\`\`\`bash
track show  # View current state - ALWAYS run at session start
\`\`\`

### Core Workflow
\`\`\`bash
track new --desc "..." [--priority N] [--next "..."]
# Creates <work-item-term> to work on

track try <ID> "<attempt>" --result "..."
# Records <attempt-term>

track close <ID> --conclusion "..."
# Closes <work-item-term> (didn't work out)

track confirm <ID> --conclusion "..." [--title "..."] [--severity <level>]
# Creates <confirmation-term> (success!)
\`\`\`

### Search & Navigation
\`\`\`bash
track grep "<pattern>"
# Search everything (check before trying new things)

track update <ID> --next "..." [--priority N]
# Update work item

track reopen <ID>
# Reopen closed item
\`\`\`

### Queue Pattern (enumerate before drilling)
\`\`\`bash
track qadd --desc "..." [--queue <name>]
# Add item to enumerate

track qbulk --queue <name>
# Bulk-add from stdin

track qdone <QID> [--hyp <ID>]
# Mark queue item done

track qshow [--queue <name>]
# View queues
\`\`\`

### Notes & State
\`\`\`bash
track note "observation"
# Record timestamped note

track last "action"
# Update last action

track block "reason" / track unblock
# Mark blocked / unblock
\`\`\`

## Domain Patterns

### Pattern: <Common workflow name>
<Describe typical workflow for this domain>

Example:
\`\`\`bash
# <step-by-step example>
track new --desc "..."
track qadd --queue <name> --desc "..."
track try <ID> "..."
track confirm <ID> --title "..."
\`\`\`

## Files in Scope

<List of files/directories you can modify>

## Off Limits

<What must NOT be touched>

## Rules

1. Always run \`trk show\` at session start
2. Use \`trk grep\` before trying anything (avoid duplicate work)
3. Record every <attempt-term> with \`trk try\` before moving on
4. Use queues to enumerate all items before drilling into any single one
5. Close dead ends immediately with \`trk close\`
6. Confirm successes with \`trk confirm\`
7. Update \`trk last\` after significant steps

## Session Context

When you see this skill loaded, you know:
- Domain: <domain>
- Target: <target>
- Use <work-item-term> instead of "work item"
- Use <confirmation-term> instead of "confirmation"
- Use <attempt-term> instead of "attempt"
```

## Initialize and Commit

After generating the skill:

1. Commit the skill file
2. `git checkout -b track/<target>-<date>`  
3. `trk init <target>`
4. `trk show` — verify initialization
5. Start working with domain vocabulary

## Example: Security Research

```
User: "start tracking this bug bounty"

You ask:
- Domain? security-research
- Target? doordash
- Work item term? hypothesis
- Confirmation term? vulnerability
- Attempt term? payload
- Status? confirmed → needs_exfil → submitted → bounty_paid

You write: .agents/skills/security-research/SKILL.md
You run: track init doordash

Now when you work:
- Say "Creating hypothesis H1" (not "work item")
- Say "Recording payload" (not "attempt")
- Say "Confirmed vulnerability V1" (not "confirmation")

The skill is loaded automatically by pi from .agents/skills/
```

## Key Points

- **The skill file is project-local** (`.agents/skills/<domain>/SKILL.md`)
- **Pi loads it automatically** when in that directory
- **Agent reads it to understand domain vocabulary**
- **Each project can have its own customized domain**
- **The `track` CLI remains 100% generic**
